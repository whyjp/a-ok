워커 프로파일
두 가지 출처를 함께 보여준다 — (1) workerctl profiles 가 DB 에 등록한 워커 실행 프로파일, (2) ~/AppData/Local/hermes/ 디스크에서 디스커버한 Hermes Agent 프로파일 (default + profiles/<name>/). DB 가 비어 있어도 hermes 측 프로파일은 항상 표시된다.

hermes 세션
~/AppData/Local/hermes/profiles/<name>/sessions/session_*.json 으로부터 동기화돼 hermes_agent_sessions 테이블에 영속화된 Hermes Agent 의 턴-바이-턴 세션. claude run 이 이 세션에서 발사된 경우 hermes_runs.hermes_session_id 링크로 자식 spawn 을 묶어 보여준다. FE 는 더 이상 디스크를 읽지 않으며 DB 가 유일한 출처다 — workerctl sessions sync-hermes 또는 BFF 의 60s 주기 워커가 동기화한다.

hermes 스폰 claude 세션
단일 hermes_sessions ledger 중 name 또는 마지막 run name 이 a-ok: prefix 로 시작하는 행 만 표시한다 — 즉 a-ok worker-control 디스패처(scripts/projects.py::_run_name()) 가 발사한 claude 세션. claude /rename 으로도 이 prefix 는 사라지지 않는다. claude -p 여부는 별도 컬럼으로만 표시 — 사람이 직접 -p 를 호출할 수도 있으므로 spawn 시그널이 아니다. 추가 디스패처 prefix 가 있으면 WORKER_CONTROL_EXTRA_SPAWN_PREFIXES env 로 등록.

관리 대상 프로젝트
workerctl projects scan 결과. owned_work (D:/work-github) 는 워커 기동/PR 가능, public_reference (D:/github) 는 read-only — 세션 시작이 정책으로 차단된다.
#이름워크스페이스정책 git브랜치경로마지막 스캔
정책: D:/work-github (owned_work, 편집/PR 가능) · D:/github (public_reference, read-only · 워커 기동 금지). 워커는 항상 claude --permission-mode auto 로 기동되며 claude -p 는 정책상 사용되지 않는다.
